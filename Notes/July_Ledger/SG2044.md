## 2. 资料依据和测试边界

SOPHGO 官方 `sophgo-doc` 仓库包含 SG2044 EVB 的 Ubuntu 镜像安装、BIOS 设置、BSP 构建、SPI Nor Flash 启动和 Boot Flow 等 HowTo 文档。公开论文中，SG2044 被描述为面向 HPC 的 64 核 RISC-V 处理器，核心为 C920v2，支持 RVV 1.0。本文只覆盖通用 Linux 用户态 C/C++ 编译运行能力，不覆盖 NUMA、PCIe、BMC、集群互联、加速库或专用服务器固件验证。

资料来源：

| 来源                                                                                                         | 关键用途                                                       |
| ---------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| SOPHGO 官方 `sophgo-doc/SG2044`                                                                              | SG2044 EVB 官方 HowTo 资料，覆盖 Ubuntu 镜像安装、BIOS、BSP、Boot Flow 等 |
| arXiv 论文 `Is RISC-V ready for High Performance Computing? Evaluating the 64-core Sophon SG2044 RISC-V CPU` | 确认 SG2044、64 核 C920v2、RVV 1.0、Linux/编译器生态评估背景              |


测试边界：

- 默认测试对象为 SG2044 EVB，且测试在板端 64 位 Linux 环境中原生执行。
- 如果实际只拿到远程服务器、QEMU/模拟器、FPGA 原型或非 EVB 平台，必须在测试记录中单独说明，不得按 SG2044 EVB 交付。
- Hello World 只验证基础编译运行闭环。
- CoreMark 承担“尽可能打开目标板用户态可用扩展”的验证。
- 公开资料只能证明资料存在，不能替代板端运行证据。
- QEMU 只能用于预验证 RuyiSDK、GCC/LLVM 命令、Hello World 和 CoreMark 脚本；不能模拟 SG2044 EVB 的 64 核 C920v2、BIOS/Boot Flow、BSP、NUMA/HPC 拓扑和板级外设，也不能作为交付证据。

## 3. 扩展参数依据

SG2044 公开资料确认的保守 ISA 方向为：

```text
RV64GCV
```

因此 CoreMark 起始参数使用：

```text
-march=rv64gcv -mabi=lp64d
```

到货后必须在 SG2044 板端执行以下命令，确认实际用户态 ISA：

```bash
uname -a
cat /etc/os-release
lscpu
cat /proc/cpuinfo
```

如果 `/proc/cpuinfo` 或 `lscpu` 输出列出更多用户态扩展，可在工具链支持的前提下补充到 `-march`。如果包含 supervisor-level 或 hypervisor-level 扩展，不应直接放入普通用户态 CoreMark 编译参数。

使用 `-mcpu` 前必须确认当前工具链支持对应 CPU 名称。SG2044/C920v2 不应直接沿用旧 SG2042/C920 的 `-mcpu=xt-c920`，除非 GCC/LLVM 支持列表明确表明该参数适用于当前 CPU。

GCC 支持检查：

```bash
riscv64-plct-linux-gnu-gcc --target-help | grep -A80 "Known valid arguments for -mcpu" || true
riscv64-plct-linux-gnu-gcc -mcpu=help || true
```

LLVM 支持检查：

```bash
clang --target=riscv64-linux-gnu --print-supported-cpus || true
```

CoreMark 参数回退顺序：

```text
rv64gcv
rv64gc
```

如实际板端 ISA 显示 `zba`、`zbb`、`zicntr`、`zicsr`、`zifencei`、`zihpm` 等扩展且工具链接受，可在 `rv64gcv` 基础上追加；失败时记录错误并回退。

## 4. 启动准备和安装日志要求

官方文档列出的前置材料：

| 项目 | 要求 | 日志记录 |
|---|---|---|
| 主机 | Linux OS 电脑 | 记录发行版、内核、`lsblk`、写卡工具版本 |
| 目标板 | SG2044 EVB | 记录实物版本、连接方式、供电方式 |
| 启动介质 | 16GB 或以上 microSD 卡、microSD 读卡器 | 记录容量、设备名，写卡前必须确认目标块设备 |
| 可选存储 | NVMe SSD | 如果使用 NVMe + microSD 组合启动，必须记录根文件系统实际所在设备 |
| 串口线 | Type-C USB 到 Type-A USB 串口线 | 记录串口设备名、波特率、从上电开始抓日志 |

镜像来源按官方文档二选一记录：

1. 直接下载 Ubuntu 镜像模板：官方文档指向 `sophgo/bootloader-riscv` release `sg2044-v0.1`。
2. 按 `How to Build SG2044 BSP.rst` 自行构建，得到 `install/soc_sg2044/single_chip/ubuntu-sophgo.img`。

如果使用下载的模板，官方文档说明模板中缺少部分 BSP 内容，需要构建 SG2044 BSP 并把固件补入 EFI 分区。日志中必须记录 BSP 仓库、分支、commit 和以下动作：

```bash
CHIP=sg2044
source bootloader-riscv/scripts/envsetup.sh
build_rv_zsbl
build_rv_sbi
build_rv_edk2
build_rv_kernel
sudo mount /dev/<sdX1> /mnt/EFI
sudo mkdir -p /mnt/EFI/riscv64
sudo cp install/soc_sg2044/single_chip/firmware/* /mnt/EFI/riscv64/
sudo sync
```

写入 microSD 时可使用 balenaEtcher 或 `dd`。如果用 `dd`，必须先记录 `lsblk` 并人工确认目标盘，避免写错盘：

```bash
lsblk
sudo dd if=ubuntu-sophgo.img of=/dev/<sdX> bs=1024M status=progress
sync
lsblk
```

如需要扩容 microSD 最后一个 root 分区，记录 `fdisk`、`e2fsck` 和 `resize2fs` 输出；这一步是可选项，不能作为是否完成启动的必要条件。

从 microSD 启动时，按官方文档记录：

1. 将 microSD 插入 SG2044 EVB。
2. 连接串口线到主机并开始记录串口日志。
3. 给 EVB 上电。
4. 登录用户名和密码均为 `ubuntu`。
5. 任何需要管理员权限的操作使用 `sudo`。

启动链证据可按官方 Boot Flow 文档记录为：

```text
BootROM -> FSBL -> ZSBL -> OpenSBI -> EDK II -> GRUB 2 -> Linux
```

官方 Boot Flow 文档还列出支持的 Linux 发行版为 Ubuntu 24.04.1 和 openEuler 24.03，支持的 Linux Kernel 版本为 6.12。实际测试报告必须以板端 `cat /etc/os-release` 和 `uname -a` 为准；如果和官方 Boot Flow 文档不一致，应标为“待验证”并记录差异。

如果采用 NVMe SSD + microSD 组合启动，官方文档要求额外处理：

- 将 `ubuntu-sophgo.img` 写入 NVMe。
- 如需要，扩容 NVMe root 分区。
- 删除 microSD 卡上的 root 分区，避免 microSD 与 NVMe 具有相同 label。
- 重启后用 `findmnt /`、`lsblk -f` 确认根文件系统确实来自 NVMe。

只要写卡、启动或登录任一环节缺失日志，就不得进入“已实测”状态。
