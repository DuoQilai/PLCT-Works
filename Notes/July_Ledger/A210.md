## 2. 资料依据和测试边界

知合计算官方文档中心提供 `A210 产品规格书`，文档版本为 `2.9.0`。该规格书记录 A210 为 8 核 64 位 RISC-V 处理器，CPU 组合为 `4xC920 + 4xC908`；C920 和 C908 均标注支持 RISC-V `64GCV`，并记录 128 位矢量寄存器位宽、FP16/BF16/FP32/FP64 与 INT8/INT16/INT32/INT64 矢量能力。本文只把这些信息作为编译参数和测试准备依据，不把规格书等同于板端实测。

同一官方文档树下提供 `镜像烧录指南`、`SDK 快速上手` 和 `硬件平台` 入口。烧录指南按型号区分 A210 SODIMM V1、A210 SODIMM V2、A210 SODIMM V3、A210 算力棒；SDK 快速上手列出 `a210-evb` 的 Buildroot 和 Debian 预编译镜像类型；硬件平台入口包含核心板用户指南、底板用户指南、底板原理图、硬件设计指南和 PINMUX。实际测试前必须先确认手上板卡属于哪个型号。

公开报道可作为补充背景，但不再作为 A210 主要资料依据。本文不覆盖 NPU、AI SDK、图形、多媒体、硬件原理图审查或多 Die 级联专项测试。

测试边界：

- 默认 A210 开发板可启动 64 位 Linux；官方快速上手已列出 Buildroot / Debian 镜像类型。如果实际为 FPGA 原型、仿真环境或远程样机，应单独标注。
- 必须先确认实际板型是 A210 SODIMM V1/V2/V3、A210 算力棒，还是其他 A210 平台，并选择对应官方烧录指南。
- 使用官方烧录指南时，必须记录镜像来源、checksum、fastboot 工具版本、串口波特率、登录账号和刷写日志。
- 官方 SDK 快速上手只证明存在 Buildroot / Debian 预编译镜像类型；仍需记录实际镜像文件、下载来源、checksum 和刷写日志。
- Hello World 只验证 GCC/LLVM 基础编译和板端运行闭环。
- CoreMark 必须带 `-march` 或 `-mcpu`，但当前只使用可解释的保守起始参数。
- 若厂商提供完整 ISA/SDK 参数，应优先采用厂商资料，同时保留工具链支持证据。
- QEMU 只能用于预验证 Debian/Buildroot 风格 RV64 Linux 用户态编译运行脚本；不能模拟 A210 SODIMM/算力棒整板、刷写、C920+C908 异构拓扑和外设，也不能作为交付实测。

## 3. 扩展参数确认原则

A210 官方规格书记录 C920/C908 均支持 RISC-V `64GCV`，因此首次 CoreMark 可以使用 `rv64gcv/lp64d` 作为起始参数。但最终参数仍要以板端 ISA 和工具链支持为准，按以下顺序执行：

1. 板端记录实际 ISA：

```bash
uname -a
cat /etc/os-release
lscpu
cat /proc/cpuinfo
```

2. 检查工具链支持：

```bash
riscv64-plct-linux-gnu-gcc --target-help | grep -A80 "Known valid arguments for -mcpu" || true
riscv64-plct-linux-gnu-gcc -mcpu=help || true
clang --target=riscv64-linux-gnu --print-supported-cpus || true
```

3. 在没有更完整证据前，CoreMark 起始参数使用：

```text
-march=rv64gcv -mabi=lp64d
```

4. 如果工具链或板端不支持向量扩展，回退到：

```text
-march=rv64gc -mabi=lp64d
```

注意：

- 官方规格书中的 `64GCV` 可支持 `rv64gcv` 起始测试，但 BF16、加密、虚拟化、TEE、NPU 等能力不能自动等价为完整用户态 `-march` 字符串。
- CoreMark 是整数基准测试，不能因为 BF16/AI 算力资料存在就推断 CoreMark 分数。
- 如果厂商提供 `-mcpu=a210` 或类似参数，只有在 GCC/LLVM 支持检查中存在该 CPU 时才能使用。

烧录主机准备：

| 主机系统 | 官方工具要求 | 日志要求 |
|---|---|---|
| Windows | 安装 Android SDK Platform-Tools `platform-tools-r34.0.5-windows.zip` 和 Google USB Driver `usb_driver_r13-windows.zip`；配置 `platform-tools` 到 `Path` | 截图或文字记录驱动安装结果；保存 `fastboot --version` 输出；保存设备管理器中 fastboot/USB 设备识别情况 |

镜像和 checksum 记录：

```bash
sha256sum <image-or-package> || md5sum <image-or-package>
ls -lh <image-or-package>
```

如果官方镜像只提供 MD5，则记录 MD5；如果同时有 SHA256，应优先记录 SHA256。无论使用 Buildroot 还是 Debian，都必须记录镜像来源页面、文件名、版本、checksum 文本和本地校验输出。

进入烧录模式后先验证主机识别：

```bash
fastboot --version
fastboot devices
```

烧录必须优先执行镜像包内官方脚本或官方文档指定命令；不要在没有官方分区表依据时手写分区烧录命令。日志中至少应能看到这些证据：

- `fastboot devices` 能识别设备。
- GPT/loader/boot/system/app/home 等分区按官方脚本顺序写入。
- 每个关键分区出现 `Sending`、`Writing` 和 `OKAY`。
- `system_a` 如出现 sparse 分片，应记录全部分片序号和耗时。
- 最终出现官方示例中的 `Images flashed success` 或等价成功标记。
- 如任一分区 `FAILED`、超时或设备断开，本轮标记为“测试失败”，不得继续写为已启动。

重启进入系统后继续记录：

```bash
uname -a
cat /etc/os-release
ip addr
whoami
```

官方烧录指南中的 Buildroot 示例以 `root` 登录，且说明 `root` 默认未配置密码；如需要 SSH，必须先执行 `passwd` 设置密码。测试报告只记录“已按交付策略修改密码”，不要在文档中明文保存长期密码。
